# 工坊物栈开发规范（AI 开发用）

## 1. 项目定位

“工坊物栈”是面向个人工坊的轻量化资源管理系统，用于管理元器件、3D 打印耗材、工具、备用材料、线材及其他物品。

本项目不是工厂库存系统，不做严格入库单、出库单、审批、采购、供应商管理、复杂盘点等企业流程。开发时优先保证：

```text
快速记录
快速查找
快速修改
位置清楚
照片和附件完整
API / CLI 可调用
后续模块可扩展
```

---

## 2. 总体架构规范

系统采用前后端分离架构：

```text
Vue 前端
   ↓
REST API
   ↓
FastAPI 后端
   ↓
Service 业务层
   ↓
Repository 数据访问层
   ↓
SQLite + 本地文件系统
```

必须遵守：

```text
Web 前端调用 API
CLI 调用 API
外部模块调用 API
任何客户端都不得直接访问数据库
```

CLI、AI、扫码、NFC、标签打印、位置地图等后续模块都必须通过 API 接入。

---

## 3. 后端分层规范

后端必须按以下职责分层：

```text
api/            只处理路由、请求参数、响应返回
services/       处理业务逻辑和业务校验
repositories/   处理数据库读写
models/         定义数据库表结构
schemas/        定义请求和响应数据结构
core/           配置、数据库、安全、统一响应、错误处理
utils/          通用工具函数
```

禁止：

```text
在 Router 中写复杂业务逻辑
在 Service 中直接拼复杂 SQL
在前端实现后端应该负责的业务规则
让 CLI 直连 SQLite
```

推荐调用链：

```text
Router → Service → Repository → Database / Files
```

---

## 4. API 设计规范

### 4.1 API First

所有核心业务能力必须先设计 API。Web 前端和 CLI 都只是 API 客户端。

### 4.2 按业务动作设计 API

API 对应业务动作，不对应前端按钮。

推荐：

```http
POST /api/items/{id}/move
POST /api/items/{id}/use
POST /api/items/{id}/adjust
POST /api/items/{id}/mark-restock
```

不推荐：

```http
POST /api/do
POST /api/action
POST /api/update_data
```

### 4.3 HTTP 方法约定

| 方法 | 用途 |
|---|---|
| `GET` | 查询 |
| `POST` | 创建资源或执行业务动作 |
| `PATCH` | 局部修改 |
| `DELETE` | 删除或归档 |

### 4.4 统一返回格式

成功返回：

```json
{
  "success": true,
  "data": {},
  "message": "ok"
}
```

失败返回：

```json
{
  "success": false,
  "error": {
    "code": "ITEM_NOT_FOUND",
    "message": "物品不存在"
  }
}
```

错误码使用大写蛇形命名，例如：

```text
ITEM_NOT_FOUND
LOCATION_NOT_FOUND
CATEGORY_NOT_FOUND
INVALID_TOKEN
VALIDATION_ERROR
DUPLICATE_CODE
BACKUP_FAILED
UPLOAD_FAILED
```

## 5. CLI 设计规范

CLI 命令名建议为：

```bash
stash
```

命令格式：

```bash
stash <resource> <action> [options]
```

示例：

```bash
stash item list
stash item show FIL-000001
stash item move FIL-000001 --to WS.CAB-A.S02.G03
stash item use FIL-000001 --amount 0.12 --unit kg
stash location tree
stash backup create
```

必须遵守：

```text
CLI 只能调用 API
CLI 不得直连数据库
查询类命令必须支持 --json
错误输出应清晰
命令命名保持一致
```

JSON 输出示例：

```bash
stash search PLA --json
stash item show FIL-000001 --json
```

## 6. 数据库设计规范

### 6.1 通用主表 + 自定义属性

不要为每类物品建立独立大表。采用：

```text
items                    物品主表
attribute_definitions    分类字段模板
item_attribute_values    物品属性值
```

这样方便后期扩展新材料类型。

### 6.2 稳定编号不用中文

中文只用于显示名称。

物品编号示例：

```text
ELE-000001
FIL-000001
TOOL-000001
MAT-000001
CAB-000001
OTH-000001
```

位置编号示例：

```text
WS.CAB-A.S02.G03
WS.DRY-A.G01
WS.TOOL-WALL.H07
```

稳定编号只允许：

```text
A-Z
0-9
-
_
.
```

### 6.3 位置系统边界

核心系统只管理逻辑位置：

```text
位置名称
层级关系
位置 code
完整 full_code
位置下物品查询
```

核心系统不管理：

```text
机械臂坐标
抓取点
运动规划
真实地图坐标
自动化设备参数
```

后续模块通过 `location.full_code` 自行关联扩展数据。

### 6.4 删除策略

物品优先使用软删除 / 归档：

```text
is_archived = true
```

不要默认物理删除，避免误删数据和附件。

## 7. 文件与附件规范

图片和附件不得存入数据库。

数据库只保存文件元信息：

```text
原始文件名
存储文件名
文件路径
MIME 类型
文件大小
所属物品
上传时间
```

实际文件存储目录：

```text
data/
├── uploads/
│   ├── images/
│   ├── attachments/
│   └── thumbnails/
└── backups/
```

上传文件必须重命名保存，避免中文路径、重名和特殊字符问题。

物品列表和卡片视图应使用缩略图，详情页再加载原图。

---

## 8. 搜索规范

搜索是核心体验，必须优先保证可用。

搜索范围包括：

```text
名称
编号
分类
位置
标签
别名
备注
自定义属性
附件名
```

必须支持不完整输入和模糊匹配，例如：

```text
esp     → ESP32、ESP8266
pla     → 黑色 PLA、白色 PLA+
m3螺    → M3 螺丝、M3 热熔螺母
开发板  → ESP32 开发板、Arduino 板
```

第一版可使用：

```text
SQLite LIKE
标签匹配
别名匹配
属性值匹配
```

后期可升级：

```text
SQLite FTS5
拼音搜索
中文分词
AI 语义搜索
向量搜索
```

## 9. 认证与安全规范

认证通过配置控制：

```env
AUTH_LOGIN_ENABLED=false
API_TOKEN_ENABLED=true
API_TOKEN_REQUIRE_ALL=true
```

要求：

```text
网页登录可选
API Token 必须保留
CLI 和外部模块必须使用 Token
Token 通过 Authorization: Bearer <token> 传递
数据库只保存 Token 哈希，不保存明文 Token
```

不要把密钥、Token、绝对路径写死在代码中。

---

## 10. 日志与记录规范

所有会修改数据的业务动作都应记录来源。

来源 `source` 可取值：

```text
web
cli
api
ai
scanner
nfc
label_printer
system
```

应记录：

```text
操作类型
操作内容
关联物品
数量变化
变化后数量
来源 source
操作者 operator
附加 metadata
创建时间
```

个人系统不需要严格库存流水，但需要保留轻量操作记录。

## 11. 备份规范

备份是第一版核心功能，不能后置。

备份内容至少包括：

```text
SQLite 数据库
uploads 文件目录
配置文件
```

要求：

```text
支持手动备份
支持备份列表
支持备份恢复
恢复备份前自动创建当前快照
备份操作必须有 API 和 CLI
```

CLI 示例：

```bash
stash backup create
stash backup list
stash backup restore backup-20260514-103000
```

## 12. 前端开发规范

前端使用：

```text
Vue 3
Vite
TypeScript
Tailwind CSS
Pinia
Vue Router
```

目录建议：

```text
src/
├── api/          API 封装
├── components/   通用组件
├── views/        页面
├── stores/       状态管理
├── router/       路由
├── types/        类型定义
├── utils/        工具函数
└── assets/       静态资源
```

要求：

```text
API 调用集中放在 src/api
组件中不要散落 fetch / axios 请求
类型定义集中管理
页面负责组织组件
组件负责展示和局部交互
不要把业务规则写死在前端
```

UI 风格：

```text
个人工坊物品卡片库
不是企业仓库后台
优先搜索和快速操作
桌面端支持左侧分类 + 中间列表 + 右侧详情
手机端单独适配，不直接压缩桌面布局
```

## 13. 测试规范

第一版至少测试以下内容：

```text
创建物品
修改物品
移动位置
增加 / 使用 / 调整数量
搜索物品
上传附件
创建备份
恢复备份
Token 鉴权
位置 full_code 生成
物品 code 唯一性
```

## 15. Git 与提交规范

个人项目不需要复杂流程，建议：

```text
main    稳定版本
dev     开发版本
```

提交信息建议：

```text
feat: add item move api
fix: fix location full_code generation
refactor: split item service
docs: update api cli design
test: add backup tests
```

避免：

```text
更新
改了一下
fix bug
111
```

## 17. 新功能开发流程

新增功能前必须按以下顺序思考：

```text
1. 这个功能是不是业务动作？
2. 是否需要 API？
3. 是否需要 CLI？
4. Service 层如何设计？
5. 是否影响数据库模型？
6. 是否影响搜索？
7. 是否影响备份？
8. 是否需要记录 source？
9. 是否需要 Token 权限？
10. 是否需要更新文档？
```

不允许只改前端而绕过后端业务设计。

---

## 18. 十条强制规则

后续 AI 开发必须遵守以下规则：

```text
1. 所有核心业务能力必须有 API
2. CLI 只能调用 API，不能直连数据库
3. 业务逻辑必须放在 Service 层
4. Router 层不得堆复杂业务逻辑
5. 稳定编号不得使用中文
6. 位置以 full_code 作为外部模块关联入口
7. 文件存本地目录，数据库只存元信息
8. 所有修改操作必须记录 source
9. 查询类 CLI 必须支持 --json
10. 新增功能必须同步考虑搜索、备份和文档
```

---

## 19. AI 开发时的输出要求

当 AI 参与开发时，必须优先输出：

```text
1. 本次开发目标
2. 涉及的文件
3. 是否新增 API
4. 是否新增 CLI
5. 是否影响数据库
6. 是否影响搜索和备份
7. 实现方案
8. 代码
9. 测试建议
```

不得直接给出一大段无结构代码。

如果发现需求可能导致系统变复杂、工厂化或破坏现有架构，必须先指出问题，再给出更合理方案。
