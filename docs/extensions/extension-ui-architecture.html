扩展声明 UI：让扩展自己定义入口和配置

这套模型把“扩展要显示什么”和“主干如何渲染”分开。扩展提供 manifest、settings schema 和 action contributions；主干只维护稳定协议、配置保存、通用渲染器和 action 转发。

整体模型

扩展声明能力，主干解释声明。这样新增扩展时通常只改扩展目录，不需要每次修改主干前端页面。

扩展侧
1. Extension Manifest

描述扩展身份、版本、配置 schema、入口和可贡献的 UI action。

扩展侧
2. Settings Schema

声明配置字段类型、默认值、必填、选项和校验规则。

扩展侧
3. Action Contributions

声明按钮、菜单、批量操作等入口应该出现在哪里。

主干侧
4. 通用渲染器

读取声明并渲染设置表单、操作按钮和扩展入口。

主干侧
5. Extension API

负责扩展发现、启停、配置保存、action 转发和幂等处理。

本机侧
6. Local Config

保存当前电脑启用了哪些扩展，以及每个扩展的本地配置。

Extension Manifest

manifest 是扩展的公开说明书。它应该进 GitHub，但不应该包含真实 token、本机绝对路径、数据库路径或用户私密配置。

manifest 负责声明

主干通过它知道扩展是谁、是否兼容当前 API、需要哪些配置，以及要贡献哪些界面入口。

extensions/
  label-printer/
    extension.json
    README.md
    src/
    config.example.json

示例 extension.json

{
  "id": "label-printer",
  "name": "标签打印",
  "version": "0.1.0",
  "api_version": "0.1",
  "settings": {
    "schema": {
      "printer_name": {
        "type": "string",
        "label": "打印机名称",
        "required": true
      }
    }
  },
  "contributions": [
    {
      "place": "item.detail.actions",
      "type": "button",
      "label": "打印标签",
      "action": "print-item"
    }
  ]
}

Settings Schema

schema 把扩展配置变成可渲染的数据结构。主干前端不需要知道“标签打印”怎么写页面，只需要按字段类型生成通用表单。

渲染结果

boolean 字段渲染为开关。

对应 schema

{
  "printer_name": {
    "type": "string",
    "label": "打印机名称",
    "required": true
  },
  "paper_size": {
    "type": "select",
    "label": "标签尺寸",
    "default": "40x30",
    "options": [
      { "label": "40 x 30 mm", "value": "40x30" },
      { "label": "60 x 40 mm", "value": "60x40" }
    ]
  },
  "copies": {
    "type": "number",
    "label": "默认打印份数",
    "default": 1,
    "min": 1,
    "max": 20
  },
  "auto_print_on_create": {
    "type": "boolean",
    "label": "新增物品后自动打印",
    "default": false
  }
}

Action Contributions

contribution 描述“入口出现在哪里”和“点击后触发哪个 action”。主干页面只预留 slot，不写死具体扩展。

推荐先开放少量位置

settings.extensions
item.detail.actions
item.list.bulk_actions
tools.menu

位置越多，主干前端维护成本越高。第一版应只开放真实需要的位置。

按钮声明示例

{
  "place": "item.detail.actions",
  "type": "button",
  "label": "打印标签",
  "action": "print-item",
  "requires": ["items.read"]
}

典型数据流

用户看到的按钮不是硬编码页面的一部分，而是扩展 manifest 被主干解析后渲染出来的结果。

1
扫描扩展

后端读取 extensions/*/extension.json。

2
读取本机启用状态

读取 extensions.local.json 或本地配置表。

3
返回声明

GET /api/extensions/contributions。

4
渲染入口

ExtensionActionSlot 根据 place 渲染按钮。

5
执行 action

POST /api/extensions/{id}/actions/{action}。

6
返回结果

扩展通过 REST API 查询或写入主干数据。

边界与落地顺序

允许

✓

扩展声明配置项、按钮、菜单和批量操作入口。

✓

扩展 action 通过主干 API 执行业务逻辑。

✓

主干保存当前电脑的启用状态和本地配置。

✓

扩展启动时调用 capabilities 判断主干能力。

禁止

×

扩展直接导入 Vue 组件、Pinia store 或前端内部类型。

×

扩展直连 SQLite 或绕过 API 修改上传目录。

×

manifest 中保存真实 token、密码或本机私密路径。

×

为了单个扩展改出复杂平台能力或插件市场。

1

最小落地版本

第一版只需要 extension.json、扩展开关、本地 settings 保存、settings schema 渲染、item.detail.actions 插槽和 POST action。等第一个真实扩展跑通后，再增加工具菜单、批量操作、健康检查和复杂独立页面。
