# 工坊物栈推荐计划 Prompt：后续分阶段路线

## 1. 你的角色

你是本项目的架构顾问兼高级后端工程师。  
你的任务不是立刻改代码，而是基于当前项目定位，给出一份**务实、分阶段、不过度设计**的后续推荐计划。

项目定位：

```text
个人开发者自用
以 Web + CLI + 后续扩展模块共同调用同一后端为核心
后续模块大多由我自己或 AI 开发
我希望边用边迭代优化，而不是先搭一个很重的完整平台
```

---

## 2. 任务目标

请给出一份清晰的后续路线图，回答下面这个问题：

```text
在“先能稳定使用，再逐步扩展”的前提下，这个后端接下来应该按什么顺序演进？
```

注意：

```text
不是空泛的长期蓝图
不是理想化大架构
而是适合当前项目真实节奏的推荐计划
```

---

## 3. 输出结构要求

请按下面 4 个阶段给出建议，每个阶段都说明：

```text
这一阶段的目标
为什么这时做它
主要建设项
明确不做什么
进入下一阶段的判断标准
```

阶段划分如下：

```text
阶段 1：先修底层主干底线
阶段 2：补最小模块化能力
阶段 3：把高频流程抽成中间层工作流
阶段 4：按真实使用情况再决定是否进一步平台化
```

---

## 4. 推荐内容重点

请重点围绕这些主题给出建议：

### 4.1 阶段 1：底层主干底线

这里应聚焦：

```text
数据完整性
库存规则
迁移策略
API / CLI 契约一致性
```

---

### 4.2 阶段 2：最小模块化能力

这里应聚焦：

```text
幂等写入
source / module / operator 来源字段
轻量任务系统
统一返回结构
```

---

### 4.3 阶段 3：中间层工作流

这里应聚焦：

```text
dry-run / plan / confirm
批量导入
批量出库
Agent 安全执行流程
```

要求你明确说明：

```text
为什么中间层应该先作为“逻辑层”存在于同一个后端里
而不是一开始就拆成独立服务
```

---

### 4.4 阶段 4：按真实使用再决定是否平台化

这里应聚焦：

```text
审计查询能力增强
更稳定的 OpenAPI
Python SDK
增量同步或能力探测
更完整的任务日志
```

同时请强调：

```text
哪些东西在当前项目里很容易成为过度设计
```

---

## 5. 架构立场

请在输出中明确你的推荐立场，不要模糊：

```text
短期先做稳底层
中期再抽中间层工作流
模块默认优先调用中间层
只有少量简单只读查询可以直接调用底层查询接口
不要过早拆独立服务
```

如果你不同意其中某条，也请明确反驳并说明理由。

---

## 6. 明确不建议优先做的内容

请单独列出当前不建议优先做的东西，例如：

```text
复杂权限系统
分布式消息队列
外部搜索引擎
重型事件溯源
过早服务拆分
模块绕过 API 直连数据库
```

并说明为什么这些在当前阶段不划算。

---

## 7. 结论要求

最后请给出一句简洁明确的总结，回答：

```text
这个项目最适合的演进方式到底是什么？
```

要求这句总结能直接作为后续讨论的共识，而不是模棱两可的折中话术。
