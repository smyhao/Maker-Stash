# 工坊物栈下一步 Prompt：后端主干分步改造总入口

## 1. 你的角色

你是本项目的高级后端工程师。  
当前不要做长期平台化扩展，不要做大而全重构。  
你的任务不是把所有事情一次做完，而是按依赖顺序，分几步把后端主干补稳。

```text
把后端主干补成一个适合 Web、CLI、后续模块和 AI 调用的稳定底座
```

项目背景：

```text
个人开发者自用
Web、CLI、后续模块都会调用这个后端
后续很多模块可能由我自己或 AI 开发
禁止模块直连数据库
我希望边用边迭代，不想现在就过度设计
```

---

## 2. 本轮要覆盖的能力

本轮最终要覆盖这 6 类能力：

```text
数据完整性
幂等
审计
任务系统
dry-run / confirm
稳定 API 契约
```

但这些能力不能放进一个 Prompt 里一次做完，否则范围会失控、验证会失真、回滚成本也会变高。

因此请按下面顺序拆成串行工作包执行。

---

## 3. 推荐执行顺序

### Prompt 1：底层主干底线修正

先完成：

```text
数据完整性
库存规则收紧
归档后写操作边界
迁移方式修正
基础 API 契约收口
```

这是整个后续能力的前提。  
如果这一步没稳，后面的幂等、审计、任务和计划执行都会建立在不可靠底层上。

对应文件：

```text
工坊物栈_分步prompt_01_底层主干底线修正.md
```

---

### Prompt 2：幂等与审计基础

在 Prompt 1 完成后，再补：

```text
关键写接口幂等
source / module / operator / request_id
最小审计日志
写操作的 before / after 记录
```

如果先做任务系统而没有幂等与审计，后面批量任务和 Agent 重试会很快把数据边界搞乱。

对应文件：

```text
工坊物栈_分步prompt_02_幂等与审计基础.md
```

---

### Prompt 3：轻量任务系统与任务 API

在 Prompt 2 完成后，再补：

```text
任务提交
任务状态查询
统一任务状态
任务返回结构
面向长任务的稳定 API 契约
```

任务系统必须建立在前两步之上，否则既不可靠也不方便排错。

对应文件：

```text
工坊物栈_分步prompt_03_轻量任务系统与任务API.md
```

---

### Prompt 4：dry-run / plan / confirm 工作流

最后，在前三步稳定后，再做：

```text
plan / confirm
批量导入预演
批量出库预演
Agent 先计划再确认执行
工作流层的稳定调用约定
```

这是最依赖前置基础的一步，不适合抢跑。

对应文件：

```text
工坊物栈_分步prompt_04_dry_run_confirm工作流.md
```

---

## 4. 核心原则

执行这些 Prompt 时，始终遵守：

```text
一次只做一个合理工作包
每一步都要能单独验证
API 契约要逐步收口，而不是同时大改所有接口
不要为了“看起来架构完整”而提前做重型抽象
先让系统可稳定使用，再逐步长成中间层工作流结构
```

---

## 5. 这份总入口的用途

这份文件不是直接改代码的开发 Prompt。  
它的作用是告诉后续 AI：

```text
应该先做哪一步
为什么要这样排序
每一步的独立目标是什么
不要把所有改造混成一次大重构
```
