# 工坊物栈分步 Prompt 02：幂等与审计基础

## 1. 前提

只有在“底层主干底线修正”完成后，才执行这一步。  
如果当前系统还存在明显脏数据入口、负库存漏洞、隐式建表问题，请先回到 Prompt 01。

---

## 2. 目标

本次只做：

```text
为关键写操作补上最小可用的幂等和审计基础
```

目的不是做大而全审计平台，而是让未来模块和 AI 重试调用时不容易重复写、也能追踪是谁改了什么。

---

## 3. 必做内容

### 3.1 幂等

对关键有副作用接口增加最小可用的幂等支持。

至少覆盖：

```text
创建物品
库存 add / use / adjust
位置移动
标识绑定
```

建议支持：

```text
request_id 或 Idempotency-Key
```

要求：

```text
同一个幂等键重复提交时，不得重复创建或重复扣减
```

### 3.2 来源字段

统一最小来源信息：

```text
source
module
operator
request_id
```

要求这些字段能在关键写操作里透传和记录，而不是只存在于个别接口。

### 3.3 最小审计日志

先做轻量审计，而不是重型事件溯源。

至少记录：

```text
who
source
module
action
target_type
target_id
before
after
request_id
created_at
```

先覆盖高价值动作：

```text
物品创建与更新
库存变化
位置移动
标识绑定
```

---

## 4. 稳定 API 契约要求

本步开始，关键写接口的调用契约要逐步统一：

```text
副作用接口支持 request_id / Idempotency-Key
错误码保持稳定
来源字段命名统一
```

不要在这一阶段全面重写所有 API，只收口关键接口。

---

## 5. 测试要求

至少补这些测试：

```text
相同幂等键重复提交不会重复执行
来源字段能正确写入或透传
审计日志能记录关键 before / after
关键错误码和返回结构稳定
```

---

## 6. 明确不做

```text
不做任务系统
不做 dry-run / confirm
不做完整工作流层
不做复杂审计查询中心
```

---

## 7. 输出要求

请直接改代码，并输出：

```text
哪些接口已支持幂等
审计日志记录了哪些动作
API 契约做了哪些收口
测试结果
哪些能力留给下一步
```
