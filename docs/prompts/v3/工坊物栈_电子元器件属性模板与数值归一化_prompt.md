# 工坊物栈 v3 Prompt：电子元器件属性模板与数值归一化

## 1. 你的角色

你是本项目的高级产品工程师兼全栈工程师。

你的任务是在当前 Maker Stash / 工坊物栈代码基础上，实现“电子元器件属性模板与数值归一化”能力。  
这是为了提升电子元器件的快速录入、搜索、筛选和列表识别能力。

请先阅读项目现有结构、`README.md`、`AGENTS.md`，以及分类、属性定义、物品表单、搜索相关代码，再动手实现。  
不要新建一套脱离现有系统的电子元器件数据库模型。必须优先复用当前已有的分类、属性定义、物品属性和搜索体系。

---

## 2. 产品背景

当前项目定位是：

```text
个人工坊物品管理系统
档案优先，库存能力保留
当前阶段主要录入电子元器件，其次是工具、螺丝、耗材等
```

用户希望系统能更好地管理电子元器件，但仍然保持个人工坊工具的轻量感。  
电子元器件属性应该标准化，但不能强制所有属性都必填，也不能把分类树做得过细。

当前方向：

```text
物品名用于人眼快速识别
标准属性用于搜索、筛选、摘要和后续扩展
自定义属性用于补充个人习惯和特殊信息
```

## 3. 核心原则

### 3.1 不新建硬编码电子元器件表

不要在 `items` 表或独立电子元器件表中硬编码字段，例如：

```text
resistance
capacitance
package
voltage
current
```

这会让系统很快变重，也不适合用户自定义分类。

正确方向是：

```text
Item 是通用物品档案
Category 决定推荐哪些属性模板
AttributeDefinition 定义属性名、类型、单位、选项、归一化规则
Item attributes 保存具体属性值
```

### 3.2 模板是一次性初始建议

系统可以提供一批电子元器件推荐模板，但它们只是初始建议。

要求：

```text
用户可以修改
用户可以隐藏
用户可以追加
后续系统不要复杂合并用户改过的模板
```

首版不要实现模板在线升级、模板版本合并、系统模板覆盖用户模板等复杂能力。

### 3.3 原始输入用于显示，归一化值用于搜索

用户输入什么，界面就尽量显示什么。

例如：

```text
用户输入：10k
界面显示：10k
内部归一化：10000 ohm
```

```text
用户输入：0.1uF
界面显示：0.1uF
内部归一化：100 nF 或 1e-7 F
```

不要用归一化结果替换用户原始输入。  
用户此前已经明确要求：数字输入多少就显示多少，不要强制补小数位或改写显示格式。

## 6. 分类策略

不要把分类树拆得过细。

例如首版不建议拆成：

```text
贴片电阻
插件电阻
金属膜电阻
采样电阻
排阻
```

更推荐：

```text
分类：电阻
属性：封装、类型、安装方式
```

开发板和模块需要分开，因为两者使用场景和核心属性明显不同：

```text
开发板：可以直接烧录、接线、运行
模块：偏功能件，例如 GPS 模块、继电器模块、升压模块、传感器模块
```

## 7. 数值归一化 v1

第一阶段只做“数值型电子属性”的归一化。  
不要做所有属性的语义归一化。

### 7.1 支持单位族

首批支持：

```text
resistance: Ω, ohm, R, k, K, M
capacitance: pF, nF, uF, µF, mF, F, 三位码如 104
inductance: nH, uH, µH, mH, H
voltage: mV, V, kV
current: uA, µA, mA, A
power: mW, W
frequency: Hz, kHz, MHz, GHz
memory: KB, MB, GB
```

示例：

```text
10k / 10K / 10000 / 10000Ω -> 10000 ohm
0.1uF / 100nF / 104 -> 同一容量
1uH / 1000nH -> 同一感量
4MB / 4096KB -> 同一存储容量
```

### 7.2 归一化必须挂在属性定义上

不要全局看到字符串就解析。

错误做法：

```text
任何字段里出现 10k 都解析为 10000 ohm
任何字段里出现 104 都解析为 100nF
```

正确做法：

```text
属性定义：阻值
normalization_kind: resistance

属性定义：容量
normalization_kind: capacitance

属性定义：Flash
normalization_kind: memory
```

`104` 这类电容三位码只允许在 `capacitance` 类型属性里解析。  
在其他字段里，`104` 可能只是编号、型号或备注，不能误解析。

### 7.3 存储建议

如果现有属性存储结构允许扩展，建议至少保留：

```text
raw_value: 用户原始输入
normalized_value: 归一化数值
normalized_unit: 归一化单位
normalization_kind: 归一化类型
```

如果现有结构暂时不适合大改，也可以采用最小兼容方案，但必须满足：

```text
原始值不丢失
搜索能命中归一化等价值
未来可以支持范围筛选
```

## 8. 录入体验

物品新增/编辑表单、连续快速录入模式都应该利用属性模板。

要求：

```text
选择分类后自动展示该分类的核心属性
扩展属性默认折叠
属性都是选填，不强制用户一次填完
属性输入允许自由输入
输入时可以根据推荐值/历史值进行匹配选择
物品名中出现的关键属性，也可以录入为结构化属性
```

示例：

```text
物品名：10k 电阻
属性：阻值 = 10k
```

```text
物品名：ESP32-WROOM 开发板
属性：主控 = ESP32-WROOM
属性：Flash = 4MB
```

不要为了属性完整性拖慢快速录入。  
本功能要服务用户边整理边录入，而不是把每个物品都变成复杂建档流程。


## 10. API 要求

本功能需要补全相应 API 接口，不能只做前端本地逻辑。

优先复用现有属性定义、分类、物品接口；如果现有接口不足，增加最小必要接口。

至少应覆盖：

```text
获取分类关联的属性模板
获取属性定义中的 normalization_kind、核心/扩展属性标记、摘要规则
创建/更新物品时提交属性原始值
后端保存属性时生成归一化值
物品详情返回属性原始值和必要的归一化元数据
搜索接口支持属性原始值和归一化等价值命中
属性模板管理接口支持用户维护模板
```

如果已有 API 已经能满足其中一部分，不要重复造新接口。  
但要确保接口契约清晰，前端连续快速录入模式、普通物品表单、搜索页和未来插件都能复用。

建议补充或确认的接口能力：

```text
GET /api/categories/{id}/attribute-definitions
GET /api/attribute-definitions
POST /api/attribute-definitions
PATCH /api/attribute-definitions/{id}
POST /api/items
PATCH /api/items/{id}
GET /api/items/{id}
GET /api/items?query=...
```

具体路径以项目现有 API 风格为准，不要为了匹配这里的示例而破坏现有路由命名。
