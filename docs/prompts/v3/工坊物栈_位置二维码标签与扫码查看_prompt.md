# 工坊物栈 v3 Prompt：位置二维码标签与扫码查看

## 1. 你的角色

你是本项目的高级产品工程师兼全栈工程师。

你的任务是在当前 Maker Stash / 工坊物栈代码基础上，实现“位置二维码标签与扫码查看”能力。  
这是主干能力，不是插件能力。它服务于位置系统、格位系统、找东西和未来自动化取料，但首版只聚焦二维码、标签打印和扫码后的查看行为。

请先阅读项目现有结构、`README.md`、`AGENTS.md`，以及位置、物品、管理界面相关代码，再动手实现。  
不要脱离现有界面重做一套新设计；必须延续当前工坊物栈的布局、配色、组件密度和交互风格。

---

## 2. 产品背景

当前项目定位是：

```text
个人工坊物品管理系统
档案优先，库存能力保留
基于位置和物品档案实现找东西、补货、整理和后续扩展
```

用户已经计划给真实收纳位置贴标签，尤其是箱子、抽屉、元器件盒的小格。  
未来可能会做机械臂自动取料，因此每一个小格需要具备稳定的位置身份。

但首版不要实现机械臂能力，也不要实现坐标标定。  
首版只需要让二维码稳定绑定到位置/格位，并支持打印和扫码查看。

---

## 3. 核心原则

二维码绑定的是“位置/格位”，不是物品。

原因：

```text
物品会移动
格位相对稳定
二维码贴在物理格位上
移动物品时不应该重新打印二维码
扫码后通过系统查询当前格位里有什么
```

二维码内容采用短协议：

```text
MSLOC:<location.full_code>
```

示例：

```text
MSLOC:LOC-6F8BCE.BOX-2.A03
```

这里的 `MSLOC:` 可视为 v1 协议标识。  
首版不需要额外的版本号字段，也不需要短码系统。

## 4. 功能范围

### 4.1 主干能力

实现以下能力：

```text
位置/格位二维码生成
A4 浏览器打印版标签导出
扫码或输入二维码内容后解析 MSLOC 协议
根据位置类型打开对应详情
```

这是主干能力，应直接放进现有应用中，而不是作为扩展插件实现。

未来可以插件化的部分：

```text
热敏打印机驱动
专用标签纸模板
扫码枪高级行为
机械臂坐标标定
自动取料动作
```

这些都不是首版范围。

### 4.2 入口设计

必须提供一个管理界面入口：

```text
管理 -> 位置二维码与标签
```

该页面用于：

```text
选择容器
查看该容器下的格位
批量生成格位标签
导出/打印 A4 标签页
提供一个扫码内容测试输入框
```

同时，在位置相关页面提供上下文入口：

```text
容器详情/位置页面：打印当前容器标签
容器详情/位置页面：打印该容器下全部格位标签
格位详情面板：打印此格标签
```

不要只把入口放在管理界面。  
实际整理物品时，用户通常站在箱子前操作，位置页的上下文入口能明显降低来回切换成本。

移动端也必须提供扫码查看入口：

```text
手机顶部操作区中，应在“新增/加号”按钮旁边提供扫码按钮
扫码按钮应使用清晰的扫码/二维码图标
移动端扫码页应优先展示摄像头扫描区域
扫码成功后打开移动端可用的位置/格位详情
移动端也要支持手动输入/粘贴 MSLOC 内容作为兜底
```

不要只实现桌面端管理页。  
贴完标签后的主要使用场景很可能发生在手机端：用户站在物理箱子前，拿手机扫格位码来查看当前物品。
移动端顶部的加号已经是高频操作入口，扫码按钮放在加号旁边，可以让“新增物品”和“扫码查看”形成并列主操作。

## 5. 标签内容

遵循“最小标签原则”，标签只显示定位必要信息。

### 5.1 格位标签

格位标签显示：

```text
大号格位号：A03
小号完整位置码：LOC-6F8BCE.BOX-2.A03
二维码：内容为 MSLOC:LOC-6F8BCE.BOX-2.A03
```

不要打印物品名称。  
因为物品会移动，标签贴在格位上，标签内容不应随物品变化。

### 5.2 容器标签

容器标签显示：

```text
容器名称：箱子2
完整位置码：LOC-6F8BCE.BOX-2
二维码：内容为 MSLOC:LOC-6F8BCE.BOX-2
```

### 5.3 首版不做

首版不要实现：

```text
标签上打印当前物品名称
标签上打印数量
标签上打印库存状态
短码生成
URL 二维码
热敏打印机适配
专用标签纸尺寸库
机械臂坐标
```

## 6. 打印与导出

首版以 A4 纸为目标，使用浏览器打印能力。

要求：

```text
提供打印预览页面或打印视图
支持 A4 页面布局
支持一次打印某个容器及其全部格位
打印样式应使用 @media print 处理
标签之间要有清晰裁切边界
二维码和文字在打印视图中不能重叠
```

如果需要二维码生成库，可以引入轻量前端库，例如 `qrcode`。  
不要为了首版引入复杂打印框架。

---

## 7. 扫码解析与跳转行为

首版应支持手机端摄像头扫码。  
用户在真实整理物品时更可能拿手机扫描格位二维码，而不是使用扫码枪。

同时保留一个文本输入/粘贴二维码内容的兜底入口，方便调试和在没有摄像头权限时使用。

解析规则：

```text
输入内容以 MSLOC: 开头
取后面的 full_code
查询对应 location
根据 location 类型决定打开方式
```

手机端扫码要求：

```text
提供移动端可用的扫码入口
请求摄像头权限
扫描到二维码后立即解析 MSLOC 协议
解析成功后跳转或打开对应详情
解析失败时给出明确提示
允许用户切换到手动输入/粘贴模式
```

可以使用成熟轻量的前端扫码库。  
不要自己手写二维码识别算法。

移动端前端要求：

```text
扫码页在手机宽度下必须可用
摄像头预览区域不能被顶部栏、底部导航或按钮遮挡
扫码结果详情在手机端应清晰展示
格位详情里的“查看物品详情”按钮在手机端可点击
权限被拒绝、无摄像头、扫码失败时，应展示手动输入入口
```

### 7.1 扫格位码

如果扫到的是格位：

```text
打开格位详情面板
显示位置：箱子2 · A03
显示当前物品：10k 电阻
按钮：查看物品详情
```

如果格位为空：

```text
打开格位详情面板
显示位置：箱子2 · A03
显示当前物品：空
查看物品详情按钮不可用或不显示
```

不要在扫码结果里提供“移动物品到此处”“放入此格”“从此格取出”等写操作。  
扫码场景下误操作风险高，首版只做查看。

### 7.2 扫容器码

如果扫到的是容器：

```text
打开该容器的位图/格位布局
高亮该容器
显示其下格位状态
```

## 8. 数据与接口建议

优先复用现有位置系统：

```text
location.full_code
容器/格位层级
格位 layout/slot 数据
物品 location_id
现有位置查询 API
现有物品查询 API
```

如果现有 API 不够，可以增加最小必要接口，例如：

```text
按 full_code 查询 location
查询某个格位当前物品
查询某个容器下全部格位
```

不要改动二维码协议去适配临时接口。  
协议应以稳定位置身份为中心。

## 9. UI 要求

界面风格必须贴合当前工坊物栈：

```text
左侧导航保持现有结构
管理页中新增入口卡片或菜单项
位置页增加轻量操作按钮

按钮和图标使用项目已有风格
```

不要做成营销页、展示页或过度装饰的卡片页面。

实现完成后，至少满足：

```text
可以从管理界面进入“位置二维码与标签”

可以在手机端扫码 MSLOC:<full_code> 并打开对应位置

```

如果改动后端：

```text

```

## 11. 实现约束

请遵守：

```text
只做首版所需能力
不要引入热敏打印机支持
不要实现短码
不要把二维码绑定到物品
不要在标签上打印物品信息
不要把扫码查看做成库存移动流程
不要只做桌面端入口而遗漏手机端前端
不要重构无关模块
```

成功的首版应该是：

```text
用户能给每个箱子和小格打印二维码
贴上后，扫描任一格位都能知道这个格子当前是什么
物品移动后不需要重新打印标签
未来仍然可以扩展到专用打印机、扫码枪和机械臂
```
